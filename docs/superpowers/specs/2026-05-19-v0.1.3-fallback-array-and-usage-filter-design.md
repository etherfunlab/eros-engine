# v0.1.3 — Fallback Model Array + Usage Key Filter — Design

**Status:** Draft for review
**Date:** 2026-05-19
**Owner:** @enriquephl

## Problem

Two independent, small surface changes both touch the OpenRouter
transport layer added in v0.1.2:

1. **`model_config.toml` `fallback` is single-string only.** A
   stale-IDs incident (issue #19) showed that one fallback often is
   not enough — when OpenRouter retires *both* the primary and the
   fallback in the same wave, every task fails. Deployers need to
   chain multiple fallbacks in order of preference.
2. **`CompanionReplyResponse.usage` exposes raw OpenRouter cost
   fields to API callers.** Commercial deployers wrapping the engine
   may not want to leak their wholesale `cost` / `cost_details` to
   their own customers (margin signal). The engine should let
   deployers filter which `usage` keys leave the server while keeping
   the full payload available to internal tracing for budget /
   alerting.

Both ship in v0.1.3 as the audit-feature maturation cut (paired with
the post-#20 code review that already landed via PR #21).

## Non-Goals

- **No persistence of usage / cost.** This release does not add a
  DB table or column for OpenRouter usage. The filter operates on
  the in-memory `usage` JSON before HTTP response serialization.
- **No per-request override of the filter.** The hidden-keys list is
  deployer-set via env. Allowing clients to disable hiding would
  defeat the purpose.
- **No content-aware filtering inside `metadata` values.** Already
  out-of-scope per the v0.1.2 audit design; restated here.
- **Tracing is unaffected.** Server-side `tracing::info!` blocks keep
  `cost` and every other usage field. Operators need full visibility
  for budgeting.
- **No parallel fallback execution.** Multiple fallback models are
  tried *sequentially* in order; the first success wins. Concurrent
  fan-out is explicitly out of scope.
- **No defaults-block array.** `[defaults] fallback_model` remains
  `Option<String>`. Per-task `fallback` is where the array support
  lives.
- **No async-route usage echo.** `/message_async` still doesn't carry
  `usage` on its response; the filter applies only to the sync
  `/message` route.

## High-Level Design

```
TOML config:
  fallback = "a"                         // legacy string still works
  fallback = ["a", "b"]                  // new: ordered list
                                              │
                                              ▼
  FallbackSpec (serde untagged enum) ── normalized ──► Vec<String>
                                              │
                                              ▼
                                ResolvedModel.fallback_model: Vec<String>
                                              │
                                              ▼
                                ChatRequest.fallback_model: Vec<String>
                                              │
                                              ▼
                  OpenRouterClient::execute ── try primary, then each
                                               fallback in order;
                                               first success wins;
                                               each failure → warn log


HTTP response path (sync /message only):

  ChatResponse.usage: Option<Value> ──┐
                                       ├──► strip hidden keys
  ServerConfig.openrouter_usage_       │    in routes/companion.rs
  hidden_keys: HashSet<String>     ────┘
                                       │
                                       ▼
                       CompanionReplyResponse.usage
```

## Feature 1: Fallback Model Array

### TOML schema

Both forms accepted on `tasks.*.fallback`:

```toml
# Legacy (continues to work)
[tasks.chat_companion]
fallback = "x-ai/grok-4.1-fast"

# New
[tasks.chat_companion]
fallback = ["x-ai/grok-4.1-fast", "deepseek/deepseek-v4-flash"]
```

**Precedence between per-task and `[defaults] fallback_model`:**

| Per-task `fallback` value             | Effective `ResolvedModel.fallback_model` |
|----------------------------------------|------------------------------------------|
| absent                                 | `defaults.fallback_model` wrapped in a singleton vec, or empty if defaults is also unset |
| `fallback = ""` (empty string, legacy) | empty vec — *explicit opt-out*; suppresses defaults |
| `fallback = []` (empty array)          | empty vec — *explicit opt-out*; suppresses defaults |
| `fallback = "x"`                       | `vec!["x"]` |
| `fallback = ["a", "b"]`                | `vec!["a", "b"]`; empty entries filtered out |

Explicit empty (string or array) means *the deployer has chosen no
fallback for this task* and suppresses the defaults singleton. Absent
means *fall through to defaults*. This matches the common convention
for layered config (e.g. `cargo` profiles).

### Types (`eros-engine-llm/src/model_config.rs`)

```rust
#[derive(Debug, Clone, Deserialize)]
#[serde(untagged)]
pub enum FallbackSpec {
    Single(String),
    Multiple(Vec<String>),
}

impl FallbackSpec {
    fn into_vec(self) -> Vec<String> {
        match self {
            FallbackSpec::Single(s) if s.is_empty() => Vec::new(),
            FallbackSpec::Single(s) => vec![s],
            FallbackSpec::Multiple(v) => {
                v.into_iter().filter(|s| !s.is_empty()).collect()
            }
        }
    }
}

pub struct TaskConfig {
    pub model: String,
    pub temperature: Option<f64>,
    pub max_tokens: Option<u32>,
    #[serde(default)]
    pub description: String,
    /// Secondary models tried in order on primary failure. Accepts a
    /// single string (legacy) or an array (new). Empty = no fallback.
    #[serde(default)]
    pub fallback: Option<FallbackSpec>,    // CHANGED from Option<String>
    pub dimensions: Option<u32>,
}

pub struct DefaultConfig {
    /// Single fallback shared by tasks that don't define their own.
    /// Intentionally kept as a single string — defaults are rarely
    /// used and array semantics would needlessly complicate them.
    #[serde(default)]
    pub fallback_model: Option<String>,    // UNCHANGED
    // …other defaults unchanged
}

pub struct ResolvedModel {
    pub model: String,
    /// Ordered fallback chain. Empty = no fallback. Engineers see
    /// the field name as singular because semantically it still
    /// resolves to *one* actually-served model — multiple entries
    /// are sequentially-tried candidates, not parallel fan-out.
    pub fallback_model: Vec<String>,       // CHANGED from Option<String>
    pub temperature: f64,
    pub max_tokens: u32,
}
```

### Resolver (`ModelConfig::resolve`)

```rust
// Precedence: explicit per-task value (even if empty) wins over
// defaults. Only an *absent* per-task fallback inherits from defaults.
let fallback_model: Vec<String> = match task_cfg.and_then(|t| t.fallback.as_ref()) {
    Some(spec) => spec.clone().into_vec(),     // explicit empty stays empty
    None => self.defaults.fallback_model.iter().cloned().collect(),
};
```

Behaviour summary:

- Per-task `fallback` absent → inherit `defaults.fallback_model`
  (singleton or empty).
- Per-task `fallback = "x"` → `vec!["x"]`.
- Per-task `fallback = ["a", "b"]` → `vec!["a", "b"]`.
- Per-task `fallback = ""` or `fallback = []` → `vec![]` (explicit
  opt-out; defaults NOT consulted).

### Wire (`eros-engine-llm/src/openrouter.rs`)

```rust
pub struct ChatRequest {
    pub model: String,
    /// Ordered fallback chain (empty = no fallback). Same singular
    /// naming rationale as `ResolvedModel`.
    pub fallback_model: Vec<String>,       // CHANGED from Option<String>
    // …other fields unchanged
}
```

### Execute (retry loop)

Replace the current `execute` body (which calls `call_once` twice,
once on primary and once on `fallback_model.as_deref()`) with:

```rust
pub async fn execute(&self, req: ChatRequest) -> Result<ChatResponse, LlmError> {
    // Build the candidate chain. Empty strings are filtered out so a
    // misconfigured TOML (e.g. `model = ""` or `fallback = [""]`) is
    // caught locally as a Config error rather than producing a remote
    // 400 from OpenRouter.
    let candidates: Vec<&str> = std::iter::once(req.model.as_str())
        .chain(req.fallback_model.iter().map(String::as_str))
        .filter(|s| !s.is_empty())
        .collect();
    if candidates.is_empty() {
        return Err(LlmError::Config(
            "openrouter: no models configured (primary empty, no fallbacks)".into(),
        ));
    }

    let mut last_err: Option<LlmError> = None;
    for (i, model) in candidates.iter().enumerate() {
        match self
            .call_once(
                model,
                &req.messages,
                req.temperature,
                req.max_tokens,
                req.user.as_deref(),
                req.session_id.as_deref(),
                req.metadata.as_ref(),
            )
            .await
        {
            Ok(resp) => return Ok(resp),
            Err(e) => {
                if i == 0 {
                    tracing::warn!(
                        primary = %req.model,
                        error = %e,
                        fallbacks_remaining = candidates.len() - 1,
                        "openrouter: primary failed, trying fallbacks"
                    );
                } else {
                    tracing::warn!(
                        primary = %req.model,
                        fallback = %model,
                        fallback_index = i - 1,
                        error = %e,
                        fallbacks_remaining = candidates.len() - i - 1,
                        "openrouter: fallback failed, trying next"
                    );
                }
                last_err = Some(e);
            }
        }
    }

    Err(last_err.unwrap_or_else(|| {
        LlmError::Config("openrouter: no models configured".into())
    }))
}
```

## Feature 2: Usage Key Filter

### Env var

```bash
OPENROUTER_USAGE_HIDDEN_KEYS=cost,cost_details
```

- Comma-separated top-level keys to strip from the `usage` JSON
  object before HTTP response serialization
- Whitespace trimmed around each key; empty entries dropped
- Unset or empty → pass-through (today's behaviour)

### `ServerConfig`

```rust
pub struct ServerConfig {
    // …existing fields
    /// Top-level keys removed from `CompanionReplyResponse.usage`
    /// before the response leaves the engine. Empty = pass-through.
    /// Tracing fields are unaffected — operator observability stays
    /// intact regardless of this setting.
    pub openrouter_usage_hidden_keys: HashSet<String>,
}
```

### `ServerConfig::from_env`

```rust
let openrouter_usage_hidden_keys: HashSet<String> =
    std::env::var("OPENROUTER_USAGE_HIDDEN_KEYS")
        .unwrap_or_default()
        .split(',')
        .map(str::trim)
        .filter(|s| !s.is_empty())
        .map(String::from)
        .collect();
```

### Application point

Only inside `routes/companion.rs::send_message`, immediately before
constructing `CompanionReplyResponse`:

```rust
let mut usage = response.as_ref().and_then(|r| r.usage.clone());
if !state.config.openrouter_usage_hidden_keys.is_empty() {
    if let Some(value) = usage.as_mut() {
        if let Some(obj) = value.as_object_mut() {
            for key in &state.config.openrouter_usage_hidden_keys {
                obj.remove(key);
            }
        }
    }
}
```

The filter does *not* apply at:

- `pipeline::run`'s usage tracing call (`log_openrouter_usage`) —
  operator visibility preserved.
- `send_message_async` — does not return `usage` to clients at all.
- `dreaming` / `post_process` — same.

## Testing

### Model config (`eros-engine-llm/src/model_config.rs`)

- `fallback_spec_deserializes_from_string` — `fallback = "a"` →
  `vec!["a"]` after `into_vec()`
- `fallback_spec_deserializes_from_array` — `fallback = ["a", "b"]`
  → `vec!["a", "b"]`
- `fallback_spec_skips_empty_entries` — `fallback = ["", "a", ""]`
  → `vec!["a"]`; `fallback = ""` → `vec![]`
- `resolve_returns_empty_fallback_when_no_task_fallback_no_defaults`
- `resolve_returns_defaults_fallback_when_task_has_none` — task has
  no `fallback`, defaults has `fallback_model = "x"` →
  `vec!["x"]`
- `resolve_task_fallback_overrides_defaults` — task array wins over
  defaults singleton
- `resolve_empty_array_suppresses_defaults` — task has
  `fallback = []`, defaults has `fallback_model = "x"` →
  `vec![]` (explicit opt-out, defaults NOT used)
- `resolve_empty_string_suppresses_defaults` — task has
  `fallback = ""`, defaults has `fallback_model = "x"` →
  `vec![]` (same as above; explicit empty wins)
- `compat_fixture_locks_full_schema` — UPDATE assertions for the
  new `fallback_model` Vec type; legacy single-string TOML still
  parses
- New `compat_fixture_array_fallback` — full task with
  `fallback = [...]`

### Execute retry (`eros-engine-llm/src/openrouter.rs` wiremock)

- `execute_returns_first_success_primary` — primary 200 → returns,
  no fallbacks attempted
- `execute_falls_back_on_primary_failure` — primary 500, fallback#1
  200 → returns fallback#1's body
- `execute_walks_full_fallback_chain` — primary + fallback#1 500,
  fallback#2 200 → returns fallback#2
- `execute_returns_last_error_when_all_fail` — primary + all
  fallbacks 500 → Err containing the last status
- `execute_returns_config_err_when_chain_empty` — primary model is
  empty string and fallback chain empty → `LlmError::Config`
  *before* any HTTP call is made (wiremock should record zero
  received requests)
- `execute_skips_empty_string_candidates` — primary `""`,
  `fallback_model = vec!["x"]` → calls only `x`, never sends an
  empty model name to OpenRouter

  - `usage_hidden_keys_from_env_parses_comma_separated` —

  call and the tests don't need to mutate process env.

  - `send_message_strips_usage_keys_when_env_set` — mock OpenRouter

    `openrouter_usage_hidden_keys = {cost}`, post a message, assert

## Risks / Open Questions

1. **LLM crate API breaking change in 0.1.x patch.** Renaming
   `ChatRequest.fallback_model` from `Option<String>` to `Vec<String>`
   and `ResolvedModel.fallback_model` likewise breaks any downstream
   caller that did `req.fallback_model = Some("x")` or
   `resolved.fallback_model.as_deref()`. Cargo's strict semver for
   pre-1.0 says 0.x's `x` bump signals breaking; 0.1.2 → 0.1.3 is a
   patch and should not break. The Rust ecosystem is loose about this
   and many 0.1.x crates ship breaking patches.

   **Decision:** ship as 0.1.3 (per direction). The audit feature
   from 0.1.2 is itself still pre-stable; downstream consumers should
   read CHANGELOG before adopting 0.1.x patches. Future republish
   should be 0.2.0 if any consumer relationship hardens.

2. **`fallback_model` singular name, plural type.** `Vec<String>`
   named `fallback_model` reads slightly awkward, but plural
   (`fallback_models`) misleads readers into thinking the candidates
   run in parallel. Semantically only one fallback is ever served
   per call; the vec is a sequential chain. Spec accepts the mild
   ergonomic cost in exchange for accurate semantics.

3. **Defaults block stays singular.** `[defaults] fallback_model`
   remains `Option<String>` and is never an array. Rationale: the
   defaults block is a low-traffic compatibility shim; adding array
   support there would force two normalization paths with no real
   user demand. If demand surfaces later, add it without breaking.

4. **Empty fallback chain after env-driven hiding.** If a deployer
   strips all `usage` keys (`OPENROUTER_USAGE_HIDDEN_KEYS` covers
   every field that ever appears), the response carries an empty
   `usage` object `{}` rather than `null`. Documented behaviour;
   acceptable.

5. **OpenRouter schema drift.** New `usage` keys added by OpenRouter
   in future will pass through by default. Deployers needing to hide
   them too just add the key name to `OPENROUTER_USAGE_HIDDEN_KEYS`.
   No code change required.

6. **No DB / no metric emission.** This release intentionally does
   not expose `cost` as a Prometheus / OTel metric. Operators wanting
   that observability can derive it from the existing tracing log
   line via their log-aggregation pipeline.
